## What Was Done

Fixed the known floor-to-floor light leak in `House_Graybox.unity`. The artifact manifested as bright horizontal bands on the building exterior at the floor/ceiling junction level, and a corresponding bright stripe visible from inside on ceiling surfaces.

## Root Cause

**Geometry protrusion through exterior walls** — not a lighting/shadow issue.

The floor and ceiling cubes (GF_Floor, GF_Ceiling, SF_Floor_A/B/C, SF_Ceiling) spanned the full building footprint X[0,14] Z[0,10], matching or exceeding the exterior wall outer edges. Since these cubes are 0.1m thick, their side faces (thin 0.1m-tall strips) were visible from outside the building. These side faces were lit by interior point lights, producing bright horizontal bands.

A secondary issue was coplanar faces: GF_Ceiling and SF_Floor pieces were both centered at Y=2.7, causing Z-fighting where they overlapped.

## Hypotheses Tested and Ruled Out

| Hypothesis | Test | Result |
|-----------|------|--------|
| Z-fighting from coplanar faces | Moved SF floors up 0.05m | Bands persisted |
| Single-sided shadow casting | Set all to TwoSided | Bands persisted |
| Directional light (no shadows, intensity=1) | Disabled directional light | No change |
| Shadow bias too high | Zeroed bias on all point lights | No change |
| Ceiling tops above wall tops | Lowered ceilings flush with walls | Bands persisted |
| **Floor/ceiling rects protruding through walls** | **Inset rects to wall inner edges** | **Fixed** |

## Key Insights for Future Work

### 1. Modular Graybox Construction Rules (for new rooms in the house graph)
- **Floor/ceiling rects must be inset to wall inner edges.** Never let a floor or ceiling cube extend to or past the exterior wall surface. The side faces of these cubes are lit by interior lights and create visible bands.
- **Ceiling tops must be flush with wall tops.** If the ceiling extends above the walls, its side face is exposed from outside.
- **Adjacent floors/ceilings must overlap by ~0.05m**, never coplanar. Coplanar faces cause Z-fighting.
- **Use TwoSided shadow casting** on all floor/ceiling MeshRenderers. Single-sided casting lets light leak through thin geometry.
- **Walls should extend into/through ceiling and floor geometry** so no gaps exist at junctions.

### 2. Debugging Methodology
- **Visual artifacts in Unity are almost always geometry, not lighting.** The research doc's ordered diagnosis (geometry first, shadows second) was correct in theory but we initially focused too much on lighting hypotheses.
- **Wireframe mode + shaded closeup from multiple angles** is the fastest way to identify the specific geometry causing an artifact. Mike's screenshots from multiple angles with wireframe toggles were the key breakthrough.
- **Unity MCP `execute_code` changes are ephemeral unless the scene is saved.** Play mode reverts to saved scene state. Always save before testing in Play mode.
- **Test in Edit mode Scene View first**, not Play mode, when diagnosing visual bugs. Play mode adds networking, scene transitions, and prefab spawning that obscure the underlying issue.

### 3. Unity MCP Operational Lessons (Critical)
- **`execute_code` transform writes do NOT persist on save.** Direct `transform.position = ...` via `execute_code` does not mark objects dirty for Unity's serialization. `manage_scene save` succeeds but the values are not written to disk. **Always use `manage_gameobject`/`manage_components` for persistent changes.** This cost us a full re-application of the fix.
- **`execute_code` operates in the editor context, not Play mode runtime.** Changes to lights/transforms via `execute_code` during Play mode do not affect runtime instances.
- **CoPlay MCP vs UnityMCP**: CoPlay failed to connect in this session (port conflict); UnityMCP was reliable. Both are available but UnityMCP was the working tool.
- **Always verify `is_playing` state** before making scene changes. Edit mode changes persist when saved; Play mode changes are discarded on exit.

### 4. Architectural Implications for the House Graph Runtime
- When the house graph spawns room nodes at runtime, each room's floor/ceiling geometry must follow these inset rules relative to its wall boundaries. This is a **construction invariant** that the room prefab authoring pipeline must enforce.
- The 0.15m wall thickness is load-bearing for the inset calculation. If wall thickness changes, floor/ceiling insets must be recalculated.
